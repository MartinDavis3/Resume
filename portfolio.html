Combustion CVD

Introduction

This work was carried out at the University of Salford and was funded by the European Union Fifth Framework Programme.

I built new equipment for combustion CVD. This was used to deposit thin films of various metal oxides on different substrates by passing appropriate chemical precursors through a flame.

The Software Challenge

Develop software to allow the user to control the gas flows (propane, oxygen and nitrogen) to the burner with sufficient accuracy and reproducibility to obtain consistent results. It was also important that the user could enter physically and chemically useful parameters such as fuel-oxygen ratio and total gas flow, rather than the less meaningful individual gas flows. To achieve this, digital mass flow controllers (MFCs) were used, connected by a RS-485 interface to a computer running software that I wrote in Visual Basic.

A Problem Encountered

Commands were sent to the MFCs through the interface buffer which then needed to be switched from send to receive in order to read the reply from the MFCs. This switching needed to occur at a precise fraction of a millisecond after the software had sent the command, a delay too short to enable use of the normally available timer in Visual Basic.

The Solution I Found

I used Windows API calls to access the CPU clock, which had the resolution necessary to allow the software to switch the interface buffer with the accuracy and precision required.

Reference: Growth of Thin Films of Molybdenum and Tungsten Oxides by Combustion Chemical Vapour Deposition using Aqueous Precursor Solutions. M. J. Davis, G. Benito, D. W. Sheel, M. E. Pemble, Adv. Mater., Chem. Vap. Deposition, 2004, 10, 29.

Spectroscopic Monitoring of a Chemical Reactor

Introduction

This work was carried out at the University of Salford and was funded by the European Union BriteEuram Programme.

A technique known as coherent anti-Stokes Raman scattering (CARS) was used to enable measurement of chemical species concentrations at precise points in a hot reactor. This used a pair of high powered, pulsed lasers together with optical components, photodetectors and signal processing electronics, all interfaced to a computer.

The Software Challenge

Write software to both control the scanning of the laser wavelength and acquire the data from the signal processing electronics, ensuring that synchronisation between wavelength and signal was maintained.

A Problem Encountered

I encountered problems with the RS-232 interface used because the standard was poorly defined and communication between the different pieces of equipment did not work.

The Solution I Found

I used a breakout box on the RS-232 line to determine that a handshake line to one of the pieces of equipment needed to be crossed-over.

Reference: In-Situ Monitoring of Atmospheric Pressure Tin Oxide CVD using Coherent anti-Stokes Raman Scattering. M. J. Davis, M. E. Pemble, J. Phys. IV France, 1999, 9, Pr8-49.

Real Time Panning of Panoramic Images

Introduction

After a trip to the Grand Canyon and Northern Arizona, I had lots of panoramic images made by stitching together many individual photographs. I found that when trying to show to show these to friends and family, the large aspect ratio meant to images lost their visual impression. While software is available to create videos that show panning across panoramic images, converting lots of images in this way can be time consuming and the resulting videos take up a lot of storage space.

The Software Challenge

I wrote software in VB.NET to enable a slide show to be made and displayed, where any image with an aspect ratio greater than a certain value would be displayed as an image at full-height and then panned in real time.

A Problem Encountered

I found that panning of a full resolution image was too slow and jerky to be aesthetically acceptable.

The Solution I Found

I ran the code to produce the panning in a separate thread which I assigned a high priority. Additionally, obtaining the correct part of the image required for each frame, which is the slowest part of the process, was done in the time between frames, copying it to a temporary buffer. This buffer could then be quickly transferred to the screen at the correct frame time. Furthermore, API calls to the internal timer were made to ensure sufficient accuracy and reproducibility of frame timing in order to give smooth panning.
